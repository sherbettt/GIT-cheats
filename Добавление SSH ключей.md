После переутсновки ОС желательно закинуть старые SSH ключи.

Проблема в правах доступа к файлу приватного ключа. 
Файл `id_ed25519` имеет права `644` (`-rw-r--r--`), что означает, что его могут читать другие пользователи. 
Для приватного ключа SSH это небезопасно.

## Исправление:

### 1. **Измените права на приватный ключ:**
```bash
chmod 600 ~/.ssh/id_ed25519
```

### 2. **Также проверьте права на другие файлы:**
```bash
chmod 644 ~/.ssh/id_ed25519.pub
chmod 644 ~/.ssh/authorized_keys
chmod 600 ~/.ssh/config  # этот файл тоже должен быть защищён
```

### 3. **Теперь добавьте ключ в агент:**
```bash
ssh-add ~/.ssh/id_ed25519
```

### 4. **Проверьте, что ключ добавлен:**
```bash
ssh-add -l
```

### 5. **Проверьте подключение к GitLab:**
```bash
ssh -T git@gitlab.runtel.org
```

## Полная последовательность команд:
```bash
# Исправляем права
chmod 600 ~/.ssh/id_ed25519
chmod 644 ~/.ssh/id_ed25519.pub
chmod 600 ~/.ssh/config

# Добавляем ключ
ssh-add ~/.ssh/id_ed25519

# Проверяем
ssh-add -l

# Тестируем подключение
ssh -T git@gitlab.runtel.org

# Теперь клонируем репозиторий
git clone git@gitlab.runtel.org:runtel/runtel-eapi-build.git
```

## Почему так важно:
- **Приватный ключ (`id_ed25519`)** должен быть доступен только вам (`600` = `-rw-------`)
- **Публичный ключ (`id_ed25519.pub`)** можно иметь публично (`644` = `-rw-r--r--`)
- **Конфигурация SSH (`config`)** тоже должна быть защищена (`600`)

После исправления прав проблема с `agent refused operation` должна исчезнуть.
